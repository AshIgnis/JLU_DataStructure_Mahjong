这份文档专门说明仓库中的 `Mahjong_completed.cpp` 是如何工作的、为什么这样设计、现在已经完成了哪些部分，以及你后续如果想继续增强，应该从哪里下手。

这份实现的目标不是“复刻完整国标麻将引擎”，而是**严格贴合 NOI2014 Mahjong 题面**，在**不改动你的原始 `Mahjong.cpp`** 的前提下，提供一个**可独立编译、可独立运行、能完成合法回放和决策**的单文件版本。

## 3. 适用的题目边界

这份实现严格按 NOI2014 题面裁剪，边界如下：

1. 只使用三门数牌：
   - `B1-B9`
   - `T1-T9`
   - `W1-W9`

2. 不处理这些内容：
   - 风牌
   - 箭牌
   - 花牌
   - 吃牌 `CHI`
   - 单独的 `BUGANG` 指令
   - 完整国标番型计算
   - 8 番门槛

3. 胡牌条件只按题面：
   - 4 个句子
   - 1 对将牌

4. 杠牌允许两类真实效果：
   - 别人打牌后直接明杠
   - 自己摸到第四张形成杠，或者碰后补成杠

## 4. 为什么没有继续使用 jsoncpp

原骨架和样例程序都包含：

```cpp
#include "jsoncpp/json.h"
```

但当前仓库没有现成的 `jsoncpp` 目录或静态库配置，所以如果直接沿用这条路线，编译环境会变得不稳定。

因此我在 `Mahjong_completed.cpp` 里内置了一个**轻量 JSON 解析器**，只解析本题真正需要的字段：

1. `requests`
2. `responses`
3. `data`

它的优点是：

1. 单文件可编译。
2. 不需要额外依赖。
3. 足够覆盖题目的输入格式。

它的取舍是：

1. 不是完整 JSON 库。
2. 只适合当前这类简单对象结构。
3. `\uXXXX` 会被简化处理，不适合做通用文本系统。

对于这道题来说，这个取舍是合理的。

## 6. 牌是怎么表示的

为了避免到处拿字符串做比较，程序内部统一用整数表示牌。

映射规则如下：

1. `B1-B9 -> 0..8`
2. `T1-T9 -> 9..17`
3. `W1-W9 -> 18..26`

例如：

- `B1 -> 0`
- `B9 -> 8`
- `T1 -> 9`
- `W9 -> 26`

这样做的好处是：

1. 数组统计非常方便。
2. 判断花色、点数只要除法和取模。
3. 做递归拆牌时比字符串快很多。

对应工具函数：

- `TileStrToId`
- `TileIdToStr`
- `SuitOf`
- `RankOf`

## 7. GameState 和 PlayerState 的设计

### 7.1 PlayerState

每个玩家维护以下信息：

1. `liveCount`
   - 当前还在“活手牌”中的牌
   - 这些牌可以被打出
   - 可以参与顺子、刻子的拆分

2. `fixedCount`
   - 已经被碰/杠固定住的牌
   - 这些牌不能再打出，也不能再拆开

3. `melds`
   - 记录已经形成的碰/杠句子

4. `discards`
   - 已打出的牌序列

5. `hasHu`
   - 这个玩家是否已经胡牌退出

### 7.2 GameState

整局状态里维护：

1. `myID`
   - 自己的座位号

2. `turnID`
   - 当前应处理的是第几条 request
   - 按 `responses.size()` 计算

3. `poolRemain`
   - 每种牌理论上还剩多少张未出现
   - 初始全为 4
   - 每看见一张牌就减一

4. `requests`
   - 平台历史输入

5. `responses`
   - 自己历史输出

6. `players`
   - 4 名玩家的状态数组

7. `pendingDiscardTile / pendingDiscardWho`
   - 记录“最近一次刚被打出的牌”
   - 用于处理后续是否被别人碰/杠/胡走

## 8. 为什么状态回放要分成两步

这份实现里最关键的思想之一，是把历史回放分成：

1. `ApplyRequestEvent`
2. `ApplySelfResponse`

原因是：**judge 的事实**和**我当时尝试执行的动作**不是一回事。

### 8.1 ApplyRequestEvent 做什么

它只处理“平台明确告诉你的事实”。

例如：

1. `0 myID`
   - 记录座位

2. `1 Card...`
   - 初始化自己的 13 张手牌

3. `2 Card`
   - 自己摸到一张牌

4. `3 who PLAY Card`
   - 某个玩家打出了一张牌

5. `3 who PENG Card`
   - 某个玩家碰了这张牌

6. `3 who GANG Card`
   - 某个玩家杠了这张牌

7. `3 who HU SELF`
   - 某玩家自摸了

8. `3 who HU Card`
   - 某玩家胡了别人打出的牌

这一步的原则是：

- 只信 judge
- 不猜测策略
- 不把“我历史上响应过什么”提前写进状态

### 8.2 ApplySelfResponse 做什么

它只负责回放“我那一回合实际生效的动作”。

这里最重要的是处理题目里的优先级问题。

典型例子：

1. 别人打出 `W5`
2. 我输出了 `PENG B7`
3. 但下一条 request 是 `3 2 HU W5`

这说明：

- 我虽然尝试碰 `W5`
- 但最终是别人优先胡走了 `W5`
- 所以我的 `PENG` 根本没有成功

这就是 `IsClaimCancelledByNextHu(...)` 这类逻辑存在的原因。

如果不这样做，状态会越来越偏，后面就会出现：

1. 手牌张数错误
2. 已固定牌错误
3. 可打牌错误
4. 胡牌判断错误

## 9. pendingDiscard 的作用

程序里专门维护了：

1. `pendingDiscardTile`
2. `pendingDiscardWho`

它们表示：

- 最近一张刚被某人打出的牌是什么
- 是谁打的

这两个变量的意义是帮助程序理解：

1. 某个 `PENG/GANG/HU` 是不是在“吃掉刚才那张打出的牌”
2. 这个动作会不会让那张牌不再作为普通弃牌继续留在牌池语义里

这是回放系统比“只记手牌”更完整的一点。

## 10. 胡牌判断是怎么做的

### 10.1 总体思路

胡牌判断按题面最经典的形式：

1. 4 个句子
2. 1 对将牌

其中句子可以是：

1. 顺子
2. 刻子
3. 杠子

### 10.2 处理流程

`CanHu` 的流程大致是：

1. 先看自己已经有多少个固定句子
   - 也就是 `CountFixedMelds(me)`

2. 计算还需要多少个句子
   - `needMelds = 4 - fixedMelds`

3. 检查活牌总数是否满足
   - `liveTotal == 3 * needMelds + 2`

4. 枚举“哪一种牌作为将牌”

5. 扣掉将牌后，把剩余活牌按三门花色拆开

6. 每门花色用 `SolveOneSuit` 判断能否完全拆成若干个句子

### 10.3 SolveOneSuit 的递归逻辑

`SolveOneSuit` 的方法很标准：

1. 找到当前这门牌里最小的非零点数
2. 尝试把它当成刻子拆掉
3. 尝试把它当成顺子起点拆掉
4. 只要有一种拆法能一直成功到底，就说明该花色合法

因为这道题只有三门数牌，没有字牌和特殊番型，所以这个做法已经足够稳定。

## 11. 合法动作是怎么生成的

统一入口是：

- `GenerateLegalActions`

它不会直接“决定做什么”，而是先回答：

**当前局面下，哪些动作是规则允许的？**

### 11.1 摸牌回合

当前 request 是 `2 Card` 时，可能生成：

1. `HU`
2. `GANG Card`
3. `PLAY Card`

### 11.2 别人打牌回合

当前 request 是 `3 who PLAY Card` 时，可能生成：

1. `PASS`
2. `HU`
3. `GANG`
4. `PENG CardAfterPeng`

这里有一个重要细节：

- `PENG Card`
  - 这里的 `Card`
  - 不是被碰的那张牌
  - 而是碰完以后你打出去的那张牌

### 11.3 流局回合

当前 request 是 `4` 时，内部会生成：

1. `Target`

最后再由 `FormatAction` 输出成一串牌字符串。

## 12. 这份策略是怎么做的

这份实现不是“无脑随机打牌”，但也不是特别重的搜索型 AI。它采用的是一套**轻量启发式策略**。

### 12.1 总体原则

优先级大致是：

1. 能胡就胡
2. 先枚举所有合法动作
3. 用评分函数比较不同动作
4. 选择评分最高的那个

### 12.2 EvaluateHandShape

这个函数会给当前手牌形状一个基础分。

它综合了这些因素：

1. 如果已经能胡，给极高分
2. 距离某个合法目标胡牌型的距离
3. 已固定句子数量
4. 对子、刻子带来的收益
5. 相邻搭子、隔张搭子的收益
6. 孤张惩罚

### 12.3 EvaluateDiscard

这个函数专门回答：

**如果我现在打掉某张牌，局面会不会更舒服？**

做法是：

1. 复制状态
2. 模拟打掉这张牌
3. 调 `EvaluateHandShape`
4. 再补充一些偏好项

目前还额外考虑了：

1. 这张牌是否已经比较“老”了
2. 这张牌是不是单张孤牌

### 12.4 ChooseBestAction

这个函数统一比较：

1. `PASS`
2. `PLAY`
3. `PENG`
4. `GANG`

处理方式是：

1. 如果有 `HU`，直接返回
2. 对于 `PLAY`，用 `EvaluateDiscard`
3. 对于 `PENG/GANG`，先模拟动作后的状态，再整体打分
4. 给杠牌少量额外奖励
5. 对碰牌做轻微惩罚，避免过度碰牌

这样做的结果通常比：

1. 能碰就碰
2. 能杠就杠
3. 随机打一张

要稳定得多。

## 13. 流局目标牌型是怎么构造的

这个部分是本实现里相对“高级一点”的地方。

统一入口：

- `BuildBestTargetHand`

内部核心搜索：

- `FindBestTargetPlan`

### 13.1 为什么不能随便拼 14 张牌

因为题目要求流局时你必须输出一个**合法可胡的目标牌型**，还要按你和它之间的海明距离进行排名。

如果随便输出一个乱拼的牌型，虽然格式对了，但策略意义很差，也不利于答辩解释。

### 13.2 当前实现做了什么

它会：

1. 根据已固定的碰/杠，先确定还需要几个句子
2. 分别对三门花色搜索“这一门最适合凑成什么”
3. 枚举句子在三门里的分布方式
4. 同时枚举将牌落在哪一门
5. 选出：
   - 海明距离最小
   - 若距离相同，则剩余牌支持度更高
   的那一个目标牌型

### 13.3 这和真实最优还有什么差距

目前这版已经比“写死一个目标牌型”强很多，但仍然不是完整的全局最优。

还可以继续加强：

1. 更准确地计算海明距离
2. 纳入牌池出现顺序，估计胡牌机会值
3. 把别人的公开信息也纳入目标搜索

## 17. 现在这份实现的局限

这部分很适合你后面写报告时直接拿去讲“当前版本的不足”。

### 17.1 JSON 解析器不是通用库

它只适合当前题目的输入风格，不适合作为通用项目基础设施。

### 17.2 状态建模仍是“单边视角”

目前重点维护的是：

1. 自己的手牌正确性
2. 别人的公开信息

但没有完整恢复别人的隐藏手牌分布，这也是题目允许的常见做法。

### 17.3 策略还是启发式，不是搜索树

目前没有做：

1. 多步前瞻
2. Monte Carlo
3. Minimax
4. 大规模状态搜索

因此它的上限还可以继续抬高。

### 17.4 流局目标牌型还可以更强

现在已经比固定模板好，但还没有把“胡牌机会值”完全做进排序模型。

“本程序采用单文件分层结构，按输入解析、状态回放、规则判断、策略评估、结果输出五个层次组织，实现了 NOI2014 简化麻将 Bot 的完整交互闭环。”

### 19.2 状态恢复

“由于平台每回合重启程序，因此程序首先基于历史 `requests` 和 `responses` 重建局面，其中将 judge 确认事件与本方历史响应区分开处理，避免因碰杠未生效造成状态污染。”

### 19.3 胡牌判断

“胡牌判断采用按花色分解的递归拆牌法，枚举将牌后分别对三门数牌进行顺子/刻子可分解性验证。”

### 19.4 决策方法

“程序先生成当前全部合法动作，再通过手牌结构评分函数对弃牌、碰牌、杠牌后的状态进行比较，选择评分最高的动作输出。”

### 19.5 流局处理

“针对流局场景，程序会搜索最接近当前牌型的合法胡牌目标，而非固定输出模板牌型，从而提升海明距离排名表现。”
