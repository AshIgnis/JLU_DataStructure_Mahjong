# NOI2014 Mahjong 项目完成指南

## 1. 先说结论

这三个材料里，真正应该作为最终标准的是 `Mahjong.pdf`。  
你的项目目标不是“复刻现在这份 `Mahjong.cpp`”，而是：

1. 以 `Mahjong.pdf` 的规则和交互格式为准。
2. 以 `MahjongSample.cpp` 的输入输出方式和历史回放思路为骨架。
3. 只从 `Mahjong.cpp` 借鉴“状态表示、递归拆牌、弃牌启发式”的思想。
4. 重新写出一个适配这道 NOI2014 麻将题的 `Mahjong.cpp`。

这题更准确地说是“简化版国标麻将交互题”，不是完整的国标麻将引擎。  
原因是它只使用 `B/T/W` 三门数牌，没有风牌、箭牌、花牌，也没有完整番型和 8 番门槛。

## 2. 三份材料分别告诉了你什么

### 2.1 `Mahjong.pdf`：规则与协议的唯一权威来源

从 PDF 可以提炼出下面这些硬约束：

1. 只使用 `B1-B9`、`T1-T9`、`W1-W9`，共 27 种牌、108 张。
2. 胡牌条件是 `4 个句子 + 1 对将牌`。
3. 句子只包括：
   - 顺子
   - 刻子
   - 杠子
4. 没有“吃牌”交互，程序只需要处理：
   - `PASS`
   - `PLAY Card`
   - `PENG Card`
   - `GANG`
   - `GANG Card`
   - `HU`
   - 流局时输出目标牌型
5. 杠后要继续摸牌，因此你的程序必须支持“摸牌 -> 杠 -> 再次摸牌”的链式回合。
6. 当别人打牌后，若同时有人可胡、有人可碰/杠，则胡牌优先。
7. 流局时不能只输出 `PASS`，必须输出一个目标胡牌型，用于排名。
8. 交互格式是 JSON，不是逐行纯文本协议。

最重要的一条是：后续编码时一切以 PDF 为准，不能让前人代码反过来定义题意。

### 2.2 `MahjongSample.cpp`：这份题目的正确交互骨架

`MahjongSample.cpp` 的价值不在策略，而在“怎么和平台说话”：

1. 它按 JSON 读取 `requests` 和 `responses`，这和 PDF 一致。
2. 它每回合都会根据历史 `requests + responses` 重新恢复现场，而不是依赖进程常驻。
3. 它用“小写字母”标记已经明示、不可再打出的牌，这和 PDF 的展示约定一致。
4. 它展示了一个很关键的细节：
   - 你历史上输出过 `PENG` 或 `GANG`，不代表这个动作最终真的生效。
   - 如果同一张牌被别人优先 `HU` 了，你的碰/杠在回放时不能算进去。
5. 它已经给出了最小可用的胡牌判定思路：
   - 先按花色计数
   - 再递归拆顺子/刻子

所以你应该把它当成：

1. 协议样板
2. 状态回放样板
3. 最小胡牌检查样板

而不是当成最终策略。

### 2.3 `Mahjong.cpp`：只能借思路，不能直接复用

这份前人代码里有值得借的东西，但整体并不适配当前项目。

可以借的部分：

1. 用整数而不是字符串表示牌。
2. 用计数数组做手牌统计。
3. 给 `摸/打/碰/杠` 分别写状态更新函数。
4. 用递归拆牌结果给弃牌打分。
5. 用“模拟动作后再评分”的思路决定要不要碰、要不要杠。

不能直接用的原因：

1. 它引入了 `MahjongGB/MahjongGB.h`，但当前仓库里没有这个目录。
2. 它读取的是另一套纯文本协议，不是 PDF/sample 里的 JSON 协议。
3. 它处理了 `CHI / BUGANG / BUHUA / 风圈 / 花牌 / FJ 字牌`，而本题都不需要。
4. 它依赖完整国标算番并检查 `fan >= 8`，而本题 PDF 根本没有这一层要求。
5. 它的牌编码覆盖 `F/J`，这和本题的 27 张数牌模型不一致。

一句话总结：

`Mahjong.cpp` 适合“拆开看思路”，不适合“直接改几行继续用”。

## 3. 我建议你把项目做成什么样

推荐目标是：

1. 保留单文件实现，最终主文件仍然叫 `Mahjong.cpp`。
2. 但内部按功能分区：
   - `parser`
   - `state`
   - `rules`
   - `policy`
   - `utils`
3. 第一版先做到：
   - 协议正确
   - 状态回放正确
   - 胡牌判断正确
   - 不输出非法动作
4. 第二版再做：
   - 更好的弃牌
   - 是否碰/杠的评估
   - 更合理的流局目标牌型

这条路线比“直接啃前人完整版 `Mahjong.cpp`”更稳，也更适合答辩。

## 4. 推荐实现方案

### 4.1 先修正工程入口

你现在的工程有两个现实问题：

1. `Makefile` 默认从 `src/` 编译，但仓库里的源文件在根目录。
2. `MahjongSample.cpp` 依赖 `jsoncpp/json.h`，仓库里没有现成的 `jsoncpp` 目录。

所以第一步不是写策略，而是先把构建跑通。

建议做法：

1. 直接把最终可提交代码集中到根目录的 `Mahjong.cpp`。
2. 修改 `Makefile`，让它直接编译根目录的 `Mahjong.cpp`。
3. JSON 解析二选一：
   - 继续沿用 sample 的 `jsoncpp`
   - 改成你本地更容易使用的单头文件 JSON 库
4. 不要再沿用前人 `Mahjong.cpp` 的纯文本输入格式。

### 4.2 数据表示

建议内部不用 `string cards[108]` 做核心状态，而改成“字符串只负责输入输出，内部统一整数编码”。

最简单的编码方式：

1. `0..8` 表示 `B1..B9`
2. `9..17` 表示 `T1..T9`
3. `18..26` 表示 `W1..W9`

建议至少维护这些状态：

1. `myID`
2. `turnID`
3. `handCount[27]`
4. `fixedCount[27]`
5. `fixedMelds`
6. `cardCount`
7. `liveTileCount`
8. `poolRemain[27]`
9. `requests`
10. `responses`

这里的关键设计是把“可打出的活牌”和“已经明示冻结的牌”分开。  
你可以像 sample 一样继续用小写表示冻结牌，也可以直接拆成两个计数数组；后者更容易做判断。

### 4.3 历史回放

这是整道题最重要的基础设施。

推荐流程：

1. 读入完整 JSON。
2. 令 `turnID = responses.size()`。
3. 遍历 `0..turnID-1` 的每个历史回合。
4. 每回合同时读取：
   - `requests[i]`
   - `responses[i]`
5. 根据请求类型更新状态。
6. 如果这一回合是你行动过的回合，再把你自己的响应应用到状态上。

你至少要覆盖这几种历史情况：

1. `0 id`
   - 记录自己的编号
2. `1 card1 ... card13`
   - 初始化手牌
3. `2 card`
   - 自己摸牌
   - 然后应用自己当时的 `PLAY/GANG/HU`
4. `3 who PLAY card`
   - 别人打牌
   - 然后应用自己当时的 `PASS/PENG/GANG/HU`
5. `3 who PENG card`
   - 这是别人的事件通知，你通常只更新牌池和轮转信息
6. `3 who GANG card`
   - 同上
7. `3 who HU SELF`
   - 有人自摸，后续这名玩家不再参与
8. `3 who HU card`
   - 有人吃和别人打出的牌
9. `4`
   - 流局

必须特别处理的坑：

1. 你历史上输出了 `PENG`，但如果下一条 request 告诉你有人 `HU` 了那张牌，你的 `PENG` 实际没有成功。
2. 你历史上输出了 `GANG`，也可能因为更高优先级的 `HU` 没真正生效。
3. 回放时一定要“以 judge 后续给出的事实”为准，而不是只信自己历史 response。

这正是 `MahjongSample.cpp` 最值得继承的部分。

### 4.4 胡牌判断

这题没有风牌、箭牌，也没有七对、十三幺之类特殊型，所以胡牌判断可以做得非常干净。

推荐算法：

1. 先统计当前所有“活牌”的计数。
2. 已固定的碰/杠直接记入 `fixedMelds`。
3. 计算还需要补多少个句子：
   - `needMeld = 4 - fixedMelds`
4. 如果活牌总数不是 `3 * needMeld + 2`，直接不能胡。
5. 枚举哪一种牌作为将牌。
6. 扣掉将牌后，对三门花色分别做递归拆分：
   - 优先尝试刻子
   - 再尝试顺子
   - 或反过来，二者都可以
7. 三门都能清空则胡牌成立。

为什么这题适合按花色拆：

1. 顺子不会跨花色。
2. 只有数牌，没有字牌特殊处理。
3. 复杂度足够低，1000ms 内很安全。

### 4.5 合法动作生成

你可以专门写一个 `GenerateLegalActions()`，不要在 `main()` 里手搓分支。

按当前请求类型划分：

1. 当请求是 `2 Card`
   - 合法动作可能是：
   - `HU`
   - `GANG CardX`
   - `PLAY CardX`
2. 当请求是 `3 who PLAY Card`
   - 合法动作可能是：
   - `HU`
   - `GANG`
   - `PENG CardX`
   - `PASS`
3. 当请求是其他通知型事件
   - 一般只能 `PASS`
4. 当请求是 `4`
   - 输出目标胡牌型

这里要特别记住：

1. 本题没有 `CHI`。
2. `PENG` 之后必须立刻跟一个你要打出的牌。
3. `GANG` 对别人的打牌响应时不带牌名，对自己摸牌后的杠要带牌名。

### 4.6 出牌与碰杠策略

第一版不要追求“很强”，只追求“稳定正确且明显强于随机”。

推荐策略顺序：

1. 能胡就胡。
2. 自摸后若有四张相同牌，可以优先考虑杠。
3. 别人打牌后，若能明杠，可优先考虑杠。
4. 是否碰牌不要写成“能碰就碰”，而要做一次简单模拟。
5. 最后在所有可打出的活牌里选一张最差的牌打出。

弃牌评分可以直接借鉴前人 `Mahjong.cpp` 的思想，但要删掉无关部分，只保留这类指标：

1. 这张牌是否形成对子。
2. 它附近有没有相邻牌或隔一张的搭子。
3. 同花色整体是否容易继续做顺子。
4. 这张牌在 `poolRemain` 中还剩几张。
5. 打掉它之后，离胡牌是否更近。

最实用的做法是“枚举每个候选弃牌，模拟打出后打分”：

1. 复制当前计数
2. 扣掉候选牌
3. 计算一个形状分数
4. 选分数最高的保留方案，也就是打掉“最差”的那张牌

`Mahjong.cpp` 里递归拆牌 + 权重打分的思路就很适合迁移到这里，但要改成 27 张牌版本。

### 4.7 是否碰牌的判断

这一步最好不要偷懒。

推荐判断方式：

1. 先算“现在如果 PASS，我的手牌评分是多少”。
2. 再模拟“如果 PENG 这张牌，并打出最佳弃牌，我的评分是多少”。
3. 只有当 `PENG` 后评分更高时才碰。

这样做的优点是：

1. 很容易实现
2. 比“无脑碰”稳定得多
3. 可以直接复用你的弃牌评分函数

### 4.8 流局目标牌型

这是 sample 最弱、但项目里必须补上的部分。

流局时输出的不是一句解释，而是一个“合法胡牌牌型”。

推荐做法：

1. 把已经固定的碰/杠视为目标牌型中必须保留的句子。
2. 对剩余需要的句子数和将牌进行搜索。
3. 枚举一批合法胡牌目标。
4. 计算每个目标与当前牌型的差距。
5. 选择海明距离最小的目标输出。

如果你不想一开始就写全局最优搜索，可以分两步做：

1. 第一版：
   - 根据当前手牌的最佳拆分结果，贪心补出最接近的胡牌型
2. 第二版：
   - DFS 枚举所有可能的剩余句子和将牌，严格比较海明距离

无论怎么做，都必须保证输出的是“合法可胡的完整牌型”，而不是随便拼 14 张牌。

## 5. 一条能真正做完项目的开发顺序

### 阶段 A：先让程序按题意跑起来

1. 修 `Makefile`
2. 解决 JSON 解析依赖
3. 按 sample 的方式读 JSON
4. 实现历史回放
5. 对 `0/1/2/3/4` 五类 request 都能给出合法 response

验收标准：

1. 不崩溃
2. 不输出非法格式
3. 能完整跑完整局

### 阶段 B：把规则做对

1. 写干净的牌编码
2. 写干净的状态更新函数
3. 写胡牌判定
4. 写合法动作生成
5. 写优先级正确的响应逻辑
6. 正确处理历史上“响应记录存在，但动作未生效”的情况

验收标准：

1. 不错和
2. 不乱碰
3. 不把冻结牌打出去
4. 流局时能给出合法目标牌型

### 阶段 C：再补策略

1. 写弃牌评分
2. 写 `PENG` 前后模拟比较
3. 写 `GANG` 的简单收益判断
4. 优化流局目标牌型搜索

验收标准：

1. 明显强于随机打牌
2. 结果可解释
3. 答辩时能讲清楚每个策略指标的意义

## 6. 我最建议你保留的函数接口

你最终即使只写单文件，也建议把函数层次整理出来：

1. `void InitState()`
2. `void ReplayHistory(const Json::Value& input)`
3. `void ApplyRequest(const string& req)`
4. `void ApplySelfResponse(const string& req, const string& resp, const Json::Value& input, int idx)`
5. `bool CanHuWithTile(int tile, bool selfDraw)`
6. `bool CanPeng(int tile)`
7. `bool CanGangOnDiscard(int tile)`
8. `bool CanGangAfterDraw(int tile)`
9. `vector<Action> GenerateLegalActions(const string& currentReq)`
10. `int EvaluateDiscard(int tile)`
11. `string ChooseDiscard()`
12. `string BuildBestTargetHand()`
13. `string DecideAction(const string& currentReq)`

这样有两个好处：

1. 逻辑会比前人那份大 `main()` 清楚很多。
2. 你后面想替换策略，不需要重写协议层和状态层。

## 7. 容易踩坑的地方

1. 不要把这题误写成完整国标麻将。
   - 不需要 `F/J`
   - 不需要花牌
   - 不需要番型计算
   - 不需要 `CHI/BUGANG/BUHUA`

2. 不要直接继承前人 `Mahjong.cpp` 的输入格式。
   - 它不是这题的 JSON 协议

3. 不要只根据当前 request 决策，不回放历史。
   - 平台每回合都会重新启动程序

4. 不要把明示的冻结牌继续当成可打牌。

5. 不要假设自己历史输出的碰/杠一定成功。

6. 不要流局时随便输出一个假目标牌型。

7. 不要先做复杂策略，再补规则。
   - 正确性永远比“看起来聪明”更重要

## 8. 最后的建议

如果你想最高效地完成这门课设，我建议你采用下面的取舍：

1. 以 `MahjongSample.cpp` 为协议骨架重写 `Mahjong.cpp`。
2. 只借 `Mahjong.cpp` 的计数数组、拆牌递归和弃牌评分思路。
3. 第一版完全不碰复杂搜索，只做：
   - 正确胡牌
   - 正确碰杠
   - 一个还不错的弃牌启发式
   - 一个合法的流局目标牌型生成器
4. 稳定版出来后，再局部增强策略。

这样做出来的程序最符合这道题，也最适合你后续写报告和答辩。
